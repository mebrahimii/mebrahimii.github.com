MapReduce is the most important processing framework in Hadoop. A processing framework, also called as infrastructure framework, is a structure accordingly to what we build software. Many different software components may have the same structure or can be build over the same framework.


Many high-level data processing languages are abstractions of MapReduce, for example the systems like Pig, Hive and Spark are heavily influenced by MapReduce concepts.


Historically, Hadoop version 1 supported only MapReduce framework.


MapReduce is also a platform and language-independent programming model at the heart of most big data and NoSQL platforms.


A programming model is a pattern/format in accordance to which we write our programs.


The logic of a MapReduce application consists of a Map phase and a Reduce phase.

The early distributed computing and grid computing frameworks has a number of limitations

- Complexity in parallel programming: The early parallel programming languages allowed for sort of parallel programming structures. It may lead to very complex programming structures whose correctness could not be verified and whose testing is very difficult.


- Hardware failures: Large number of hardware components increase probability of failure in any moment of time. There were no spare components that could automatically replaced the failed ones.


- Bottlenecks in data exchange: Transfer of data between nodes in a grid is time consuming. If certain group of data blocks must visit all nodes in a grid to be compared with all group of data blocks at all other nodes then it creates a long queue of data transfers between the nodes.


Scalability problem; Linearly increasing the computational power of hardware ( fro example by adding more nodes to a grid) does not linearly increase performance of the computations. It means that some hardware is used not for the computations but for the internal management of computational processes, e.g. switch the nodes from one task to another.

The 2004 Google MapReduce white papers determined the following design goals of MapReduce:


Automatic parallelization and distribution: It should be possible to implement an application as a sequential one and such that is can be automatically parallelized and processed on a cluster of computers (distributed over many nodes in a computer cluster).


Fault tolerance: The system must be immune to failures of the nodes in a computer cluster, it leads to an idea of replication, the nodes keeping data and performing computations must be replicated, additionally failure of a node in a cluster must automatically activate a replica node.


Input/output (I/O) scheduling; The system must be able to optimize data transfers between the persistent storage devices located on each node in a computer cluster, the system must be able to optimize data transfers between the node in a computer cluster.


Status and monitoring: The system must be able to continuously report it present state and to monitor process ion of the user applications.

MapReduce framework assumes very simplistic organization of data. All data must be organized into files of key-value pairs. In such organization, each data item consists of a sequence of two data items. The first data item is called "key" and the second data item is called "value". A "key" component place a role of a name of a property and a "value" component plays a role of a value of a property. Speaking in the terms of relational data model "key" is an "attribute" in a header of a relational table and a "value" is a value of an attribute in a row of a relational table.


For example, a "key" "City" can be associated with a value "Sydney". It means that a name of a city is Sydney.


For example, a key "Date" can be associated with two dates: data of birth and optionally date of death. It means that a value may have its internal structure.


A "key" must be unique within a collection of data it is included in.


Semistructured data like for example a sequence of words organized into lines must be converted to key-value format before processing. For example, words extracted from the lines can be associated with a counter equal to the total number of occurrencies of a word in a text.


For example a text:


text to

pass to

reader


can be converted into the following sequence of key-value pairs:


text:1, to:2, pass:1, reader:1


A sample transformation procedure can be organized as the following sequence of steps:

(1) extract the words from text and assign each word with a number 1:

text:1, to:1, pass:1, to:1, reader:1


(2) sort the pairs by words

pass:1,reader:1,text:1,to:1,to:1


(3) perform summation of ones for key-value pairs that have the same key:

pass;1,reader:1,text:1to;2


A procedure explained above as a very simple example of application of MapReduce framework.

The present slide show a more sophisticated and more practical application of MapReduce framework.


In online purchasing, users sometimes abandon their shopping carts before completing the purchase. For example, at the end of purchase procedure, just before payment, we find that delivery fee is too high, or that free delivery was promised and at the end total amount of money is automatically increased by a delivery fee. We usually get upset and we abandon a shopping cart before the completion of payment procedure.


In order to improve their business, companies are usually interested to find out more about the nature of these abandoned purchases.

Assume that we collect information about the users' action in a log file.


At Map stage of MapReduce framework we convert information taken from the logs into:

(1) The final pages visited by the users.

(2) The contents of the abandoned shopping carts.

(3) The user's session transaction state.


At Reduce stage of MapReduce framework we convert information taken from the logs into:

(1) Aggregated data for the total number of abandoned carts.

(2) The most common final page visited by the users when they ended their website visit abandoning their shopping carts.


In the other word Reduce stage performs aggregation of information obtained from Map stage.

  • In order to improve their business, companies are usually interested to find out more about the nature of these abandoned purchases
  • A MapReduce job for this analysis

Hadoop implementation of MapReduce framework frees the users from the low-level communication and coordination of nodes and processes. There is no need to synchronize parallel computations.


Hadoop let programmers focus on the MapReduce implementation and a few configuration parameters.


As the data file is usually too large to be stored in a single persistent storage device of the commodity hardware, Hadoop handles the shipment of code to data fragments (data locality).


This can dramatically reduce the overhead of network transmits.


Large mounts of data are automatically distributed over the nodes of a computer cluster. Distribution is hidden from the users. The users still see a single data containes even data is located in many different places.

Why Hadoop is useful to Big Data ?


- Hadoop provides cost-effective and fault-tolerant storage (HDFS)

- Implementation of Big Data applications is scalable, it means that when more hardware can be added to the existing cluster then and there is no need to rewrite the applications

- Data that is ingested online may be interpreted at runtime

- Due to a very simple data model it provides low cost in storing unstructured and semi-structured data

- Distribution of data on the nodes of a computer cluster is fast

- Separation of programming logic and scheduling/management makes programming of Big Data Applications much simpler

- There exist multiple levels of distributed system abstractions built on the top of MapReduce framework like: Hive, Pig, Spark

- It is possible to use multi-language tooling: Java, Python: MapReduce; SQL: Hive; data-flow: Pig; Scala, Python: Spark

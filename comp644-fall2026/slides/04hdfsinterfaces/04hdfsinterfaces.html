A Hadoop cluster is deployed across a cluster of computer nodes. As Hadoop is developed in Java, all Hadoop services operate within Java Virtual Machines (JVMs) running on these cluster nodes.


Hadoop also offers a pseudo-distributed mode, where all necessary Hadoop services and their corresponding JVMs run on a single machine. In our specific setup, this machine is a Virtual Machine running either Ubuntu 14.04 or 22.04.


HDFS (Hadoop Distributed File System) provides several interfaces for interacting with the filesystem, allowing users to read, write, interrogate, and manage files. These interfaces include:

- The file system shell (Command-Line Interface): This can be accessed using either hadoop fs or hdfs dfs commands.

- Hadoop Filesystem Java API: A programmatic interface for Java applications.

- Hadoop simple Web User Interface: A web-based interface for basic monitoring and management.

- Snakebite pure Python HDFS client library: A client library specifically for Python applications to interact with HDFS

Even though a user account might be pre-configured in your virtual machine environment, it's essential to understand how HDFS user directories are managed.

Creating an HDFS User Account Directory


While the "bigdata" user account itself is assumed to be set up on the virtual machine, you need to create a corresponding home directory for this user within HDFS. This is done using the hdfs dfs -mkdir command with the -p flag, which creates parent directories if they don't already exist:


$ bin/hdfs dfs -mkdir -p /user/bigdata


This command establishes /user/bigdata as the home directory for the bigdata user within HDFS.

Creating a Folder within the HDFS Home Directory


Once your HDFS user directory is set up, you can create subfolders within it. For instance, to create an "input" folder in your HDFS home directory, you'd use:


$ hdfs dfs -mkdir input


When you execute this command from what HDFS considers your "home" directory (which, by default, is /user/, in this case /user/bigdata), it will create the input directory directly within /user/bigdata.


Viewing Folders in Your HDFS Home Directory

To verify the creation of the input folder and see other contents of your HDFS home directory, you can use the hdfs dfs -ls command without specifying a path. This command lists the contents of your current HDFS working directory, which defaults to your HDFS home directory:


$ hdfs dfs -ls

Found 1 item

drwxr-xr-x - bigdata supergroup 0 2017-07-17 16:33 input


The output confirms that an input directory exists, owned by bigdata and belonging to the supergroup. The drwxr-xr-x indicates it's a directory with read, write, and execute permissions for the owner, and read and execute permissions for the group and others.

Uploading a File to HDFS

To transfer a local file from your machine to the HDFS input folder, you use the hdfs dfs -put command. For instance, to upload a file named README.txt (which should be present in your current local directory) into the HDFS input folder, you would execute:

$ hdfs dfs -put README.txt input


This command copies README.txt from your local filesystem into the input directory within your HDFS home directory (e.g., /user/bigdata/input/README.txt).

Verifying Contents of the HDFS input Folder


To confirm that the README.txt file has been successfully uploaded to the input folder in HDFS, you can list its contents using the hdfs dfs -ls command. While the user's provided example shows an extra hdfs dfs -put README.txt input command before the ls, the ls command is what actually verifies the upload:

$ bin/hdfs dfs -ls input

-rw-r--r-- 1 bigdata supergroup 1494 2017-07-12 17:53 input/README.txt


The output confirms the presence of README.txt within the input directory. It also provides details such as file permissions (-rw-r--r--), replication factor (implied as 1 in pseudo-distributed mode), owner (bigdata), group (supergroup), file size (1494 bytes), and the last modification timestamp.


Reading a File Located in HDFS

To view the contents of a file stored in HDFS directly through the shell, you use the hdfs dfs -cat command. For example, to display the contents of README.txt from the input folder:

$ bin/hdfs dfs -cat input/README.txt

< Contents of README.txt is displayed here >


This command will stream the entire content of the README.txt file to your console.

HDFS Path Representation as a URI

In HDFS, file and directory paths are formally represented as Uniform Resource Identifiers (URIs), which include the hdfs:// prefix. This prefix indicates that the resource is located within the Hadoop Distributed File System.

For example:

hdfs://:/user/bigdata/input

This URI specifically refers to the input folder located within the /user/bigdata directory in HDFS. Here, would be the network name or IP address of your NameNode (the master node in HDFS), and is the RPC port the NameNode listens on (typically 9000).


hdfs://:/user/bigdata/input/README.txt

Similarly, this URI points directly to the README.txt file situated inside the input directory, which itself is under /user/bigdata in HDFS.


Simplified Path Notation in HDFS Shell

When you are interacting with the HDFS command-line interface (using hdfs dfs or hadoop fs), Hadoop provides a convenient simplification for path notation in its default configuration.

You can often omit the hdfs://:/user/ part of the URI.


This means that if you are the bigdata user and your current working directory in HDFS is set to /user/bigdata (which is typically the default), you can simply refer to files and directories relative to that location.


Therefore, the command:

hdfs dfs -ls input


is effectively a shorthand for the full URI:

hdfs dfs -ls hdfs://:/user/bigdata/input


Hadoop automatically resolves the relative path input to the absolute URI hdfs://:/user/bigdata/input based on the user's default HDFS home directory and the cluster's configuration.

The web interface to HDFS provides a browser-based way to interact with the Hadoop Distributed File System. It runs on the NameNode and DataNodes, offering a user-friendly alternative to the command-line interface.

Here's a breakdown of the HDFS web interface:

Accessing the Web Interface:

- NameNode UI: Typically accessible on port 9870 (for Hadoop 2.x it was 50070) of the NameNode's hostname or IP address. The URL would be something like http://:9870.

- DataNode UI: Each DataNode also runs a web server, usually on port 9864 (for Hadoop 2.x it was 50075). You can often find links to the DataNode UIs from the NameNode web interface.

- HttpFS UI: If the HttpFS service is running, it provides a REST API over HTTP and usually has a web UI on a configured port (e.g., 14000).

Snakebite: Programmatic HDFS Access from Python

Snakebite is a Python library developed by Spotify that enables Python applications to interact with Hadoop's HDFS. It offers a more Pythonic way to perform HDFS operations compared to executing shell commands.

Key Features and Details:

- Direct NameNode Communication: Snakebite communicates directly with the HDFS NameNode using protocol buffer messages. This offers several advantages:

Efficiency: Bypassing the overhead of shell command execution and parsing, Control: Fine-grained control over HDFS operations, Integration: Seamless integration within Python applications.

- Protocol Buffers: Protocol Buffers are a language-neutral, platform-neutral, extensible mechanism for serializing structured data, developed by Google. HDFS uses them for communication between clients (like Snakebite) and the NameNode.

- Pythonic Interface: Snakebite provides a set of Python classes and methods that map to HDFS operations, making it easier for Python developers to work with HDFS.

- Command-Line Interface (CLI): As highlighted, Snakebite includes its own CLI (snakebite) that mirrors many of the standard HDFS shell commands. This can be useful for quick interactions and scripting directly from the terminal using Snakebite.


Extending the Sample Application and Concepts:

1. Understanding the Output:

The output you provided for listing /user shows a Python dictionary representing the metadata of a directory named bigdata within /user. Let's break down the keys:

'file_type': 'd': Indicates that this is a directory ('f' would signify a file).

'permission': 493: The octal representation of the file permissions (equivalent to rwxr-xr-x).

'path': '/user/bigdata': The full path of the directory in HDFS.

'length': 0: For a directory, the length is typically 0. For a file, it would be the file size in bytes.

'owner': 'bigdata': The user who owns the directory.

'group': 'supergroup': The group associated with the directory.

'access_time': 0: The last access time of the directory (may not be tracked in all HDFS configurations).

'blocksize': 0: For a directory, the block size is typically 0. For a file, it's the size of each HDFS block.

This Python script provides a basic example of using the Snakebite library to retrieve and display the metadata (parameters) of a specific file, NOTICE.txt, located in the /user directory of your Hadoop Distributed File System (HDFS). Let's break down the code step by step:

1. #!/usr/bin/env python3:

This is a shebang line. It's typically the first line of a script and tells the operating system that this script should be executed using the python3 interpreter.

2. from snakebite.client import Client:

This line imports the Client class from the snakebite.client module. The Client class is the primary interface provided by Snakebite to interact with the HDFS NameNode. It allows you to perform various HDFS operations programmatically.

3. client = Client('localhost', 9000):

This line creates an instance of the Client class.

'localhost' specifies the hostname or IP address of the HDFS NameNode. In this case, it assumes the NameNode is running on the local machine.

9000 specifies the port number on which the NameNode's client RPC server is listening. This is the port Snakebite uses to communicate with the NameNode. Note: The default port for the NameNode's client RPC in newer Hadoop versions (2.x and later) is often 8020. If you are using a more recent Hadoop setup, you might need to change 9000 to 8020.

4. for x in client.ls(['/user/NOTICE.txt'])::

This line initiates a loop that iterates over the results of the client.ls() method.

client.ls(['/user/NOTICE.txt']) calls the ls() method of the Client object. The ls() method is analogous to the hdfs dfs -ls or hadoop fs -ls shell command. It takes a list of HDFS paths as arguments and returns an iterator that yields information about the files and directories at those paths.

In this specific case, we are providing a list containing a single path: '/user/NOTICE.txt'. This means Snakebite will request information about the file (or directory, if it existed) at this exact path.

The loop iterates through each item returned by the ls() method. Since we are listing a specific file, the iterator will typically yield only one item (a dictionary containing the file's metadata) if the file exists. This item is assigned to the variable x in each iteration.

5. print(x):

Inside the loop, this line prints the value of x. As seen in the output below the code, x is a Python dictionary containing various parameters (metadata) of the /user/NOTICE.txt file.


Output Explanation:


The output you provided is a Python dictionary representing the metadata of the /user/NOTICE.txt file as returned by Snakebite's ls() method:


{'file_type': 'f', 'permission': 420, 'path': '/user/NOTICE.txt', 'length': 1541, 'owner': 'bigdata','group': 'supergroup', 'block_replication': 1, 'modification_time': 1711864616929,'access_time': 1711864616715, 'blocksize': 134217728}


In summary, this script uses the Snakebite library to connect to the HDFS NameNode, requests information about the /user/NOTICE.txt file, and then prints a dictionary containing the file's metadata, such as its type, permissions, size, owner, group, replication factor, modification and access times, and block size. This demonstrates a fundamental way to programmatically access file information in HDFS using Python and Snakebite.

#!/usr/bin/env python3

from snakebite.client import Client

client.ls(['/user/NOTICE.txt']):

The ls() method is called with a list containing a single HDFS path: '/user/NOTICE.txt'.

This tells Snakebite to fetch the metadata specifically for the NOTICE.txt file within the /user directory.

The ls() method returns an iterator. If the file exists, this iterator will yield a single dictionary containing all the metadata for NOTICE.txt (like permissions, owner, size, etc., as seen in the previous explanation).

for x in ...: The loop iterates over the items yielded by client.ls(). In this case, x will be the dictionary containing the metadata for NOTICE.txt.

print(x['path'], x['file_type']):

Instead of printing the entire dictionary x, this line specifically accesses two keys from the dictionary:

x['path']: Retrieves the full HDFS path of the entry.

x['file_type']: Retrieves the type of the entry. 'f' stands for "file", and 'd' stands for "directory".

The print() function then outputs these two values, separated by a space.

Output:

/user/NOTICE.txt f: This output confirms that the entry at /user/NOTICE.txt is indeed a file (f).

This Python script demonstrates how to use the Snakebite library to copy a file from HDFS (Hadoop Distributed File System) to your local file system. This is equivalent to the hdfs dfs -copyToLocal or hadoop fs -get shell commands.


1. #!/usr/bin/env python3:

This is the shebang line, which tells the operating system to execute the script using the python3 interpreter.

2. from snakebite.client import Client:

This line imports the Client class from the snakebite.client module. The Client class is the main interface provided by Snakebite for interacting with the HDFS NameNode.

3. client = Client('localhost', 9000):

This line creates an instance of the Client class. 'localhost' specifies the hostname or IP address of your HDFS NameNode. 9000 specifies the RPC port on which the NameNode is listening for client connections. (As a reminder, for Hadoop 2.x and later, this is often 8020).

4. for x in client.copyToLocal(['/user/NOTICE.txt'],'.')::

This is the core of the script. It calls the copyToLocal() method of the Client object.

copyToLocal() takes two main arguments:

['/user/NOTICE.txt']: This is a list of one or more HDFS source paths. In this case, we are specifying that we want to copy the file named NOTICE.txt located in the /user directory of HDFS.

'.': This is the local destination path. The . (dot) signifies the current working directory where the Python script is being executed. So, the file will be copied to the same directory where you run this script.

The copyToLocal() method returns a generator. You need to iterate over this generator (even if there's only one file being copied) for the operation to execute. Each x in the loop will be a dictionary containing the result of the copy operation for a specific file.

5. print(x):

Inside the loop, this line prints the dictionary x that contains the status and details of the copy operation for each file.

5. for y in file::

This is a nested for loop. In each iteration of the outer loop, file (a chunk of data) is itself an iterable (specifically, a bytes object in Python).

This inner loop iterates through each individual byte within the current chunk of data (file).

6. print(y):

Inside the inner loop, this line prints the value of y. Since y represents a single byte from the file's content, it will be printed as an integer representing the ASCII value of that byte. This is likely not the desired output for human-readable text.

client.text(files) takes a list of HDFS file paths as input (in this case, the files list we populated in the previous loop).

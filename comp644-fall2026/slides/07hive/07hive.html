Hive is a software system that provides a tabular view of data stored in HDFS and SQL-like methods for manipulating data in HDFS. The Apache Hive project started at Facebook in 2010 to provide a high-level interface to HDFS. Contrary to Pig, Hive provides SQL-like abstractions on top of MapReduce.


A language called HQL (Hive Query Language) implements the SQL-92 standard (almost). HQL provides a tabular view of data and it can be used to access data located in HDFS. Hive frees data analysts from needing extensive Java MapReduce programming skills (though not completely). HQL statements are parsed by the Hive client and translated into a sequence of Java MapReduce operations, which are later on processed by Hadoop.


Beyond its core functionality, Hive offers several key features:


Schema on Read: Unlike traditional relational databases that enforce a schema during data loading, Hive uses a "schema on read" approach. This means the schema is applied when the data is queried, offering flexibility as new data formats or requirements emerge.


Support for Various Data Formats: Hive can work with a wide array of data formats, including text files, SequenceFiles, RCFile, ORC, and Parquet, allowing it to process diverse datasets stored in HDFS.


Extensibility: Hive is highly extensible, allowing users to write custom user-defined functions (UDFs), user-defined aggregation functions (UDAFs), and user-defined table functions (UDTFs) in Java or other languages to extend its functionality.


Integration with Other Hadoop Ecosystem Components: Hive seamlessly integrates with other components of the Hadoop ecosystem, such as Apache Tez and Apache Spark for faster execution engines, as well as Apache ZooKeeper for coordination and Apache Oozie for workflow scheduling.


Partitioning and Bucketing: Hive supports partitioning tables by column values (e.g., by date, region) and bucketing (hashing data within partitions into fixed-size buckets) to improve query performance by reducing the amount of data that needs to be scanned.


Metastore: Hive uses a metastore to store metadata about tables, partitions, and columns. This metadata can be stored in various databases like MySQL, PostgreSQL, or Derby, allowing multiple Hive clients to share the same metadata.


While Hive significantly simplifies data analysis on Hadoop by offering a familiar SQL interface, it's important to note that it was initially designed for batch processing rather than real-time queries. Its performance can vary depending on the complexity of the query and the size of the data. However, with the integration of Tez and Spark as execution engines, Hive's performance for analytical workloads has greatly improved.

The results of processing by Hadoop are returned to the client or saved in HDFS. This flexibility allows users to choose how they want to utilize the output of their HQL queries.


When results are returned to the client, it's typically for smaller datasets or for interactive analysis where the user wants to immediately see the outcome of their query. The Hive client, whether it's the command-line interface (CLI), a JDBC/ODBC connection from a BI tool, or a web interface like Hue, will display the results directly. This is convenient for debugging queries, performing quick data explorations, or generating small reports.


When results are saved in HDFS, it's usually for larger datasets or when the output of one query needs to be used as input for subsequent Hive queries, other Hadoop jobs (like Pig or Spark), or external applications.


A typical processing of HQL statemenst starts from parsing and creation of optimized [rocessing plans. Then it is followed by submission of jobs to Hadoop and monitoring of progress. Parsing and optimization of processing plans requires access to metastore that conatins information about implemenation of tables in HDFS and data disctionary.


The results of processing are recorded in HDFS.Saving results to HDFS allows for:

Persistence: The data is stored reliably and can be accessed later.

Scalability: HDFS can handle petabytes of data, so even very large result sets can be stored.

Integration: The results become part of the Hadoop ecosystem and can be easily consumed by other tools and processes.

Reusability: The output can serve as a new, derived dataset for further analysis without needing to re-run the original, potentially long-running, query.

Hive is available on all commercial distributions of Hadoop and on Hadoop installations on virtual machines.


A relational embedded database system called Derby is used by default for the implementation of Hive's metastore. However, it is possible to use other, more robust relational database systems for the implementation of the metastore, such as MySQL, PostgreSQL, or Oracle. Using an external metastore is highly recommended for production environments as it allows for multiple Hive clients to share the same metadata, provides better performance, and offers higher availability and scalability compared to the embedded Derby database.


To use Hive, Hadoop and HDFS must be "up and running." Hive relies entirely on Hadoop for distributed storage (HDFS) and distributed processing (MapReduce, Tez, or Spark). Without a functional Hadoop cluster, Hive cannot access data or execute queries.


A top-level view of data provided by Hive consists of databases and tables. This hierarchical structure is familiar to users of traditional relational database management systems:

Databases: In Hive, a database is essentially a logical grouping of tables. It helps organize data and provides a namespace for tables, preventing naming conflicts. Users can create, drop, and alter databases. When a database is created, Hive typically creates a corresponding directory in HDFS to store the data files for tables within that database.

Tables: Tables are the fundamental units for storing and querying data in Hive. They are schema-on-read, meaning the schema (column names, data types) is defined when the table is created, but the data itself is not validated against the schema until a query is executed. Hive tables can be either:

Managed Tables: Hive manages both the data and the metadata for these tables. If a managed table is dropped, both the table definition and its underlying data in HDFS are deleted.

External Tables: For external tables, Hive only manages the metadata. The data itself resides in an HDFS location specified by the user, and it is not deleted when the table is dropped. This is useful for sharing data with other Hadoop tools or for keeping data independent of Hive's lifecycle.

Metastore contains the mappings of tables to their directory locations in HDFS. This is a crucial function, as it allows Hive to know exactly where to find the data files associated with each defined table.


The Metastore is essentially a relational database that is read and written by the Hive client. When a user issues a command to CREATE TABLE, ALTER TABLE, or DROP TABLE, the Hive client interacts with the Metastore to update this metadata. Similarly, when a SELECT query is executed, the Hive client queries the Metastore to retrieve the necessary information about the table's structure and location before it can generate the MapReduce (or Tez/Spark) jobs.


Beyond just directory mappings, the Metastore also includes critical information about the input and output formats for the files represented by the table objects. For example, it stores details like:

InputFormat and OutputFormat classes: These specify how Hive should read and write data from HDFS. Examples include TextInputFormat for comma-separated values (CSV) or tab-separated values (TSV) files, SequenceFileInputFormat for binary key-value pairs, OrcInputFormat for Optimized Row Columnar (ORC) files, and ParquetInputFormat for Parquet files.

SerDes (Serialization/Deserialization) functions: These are vital components that define how data within a record is converted to and from a sequence of bytes. When Hive reads data, the Deserializer converts the raw bytes from HDFS into Hive's internal data types for processing. When Hive writes data, the Serializer converts Hive's internal data types into bytes to be stored in HDFS. Common SerDes include LazySimpleSerDe for delimited text files, OrcSerde for ORC files, and ParquetHiveSerDe for Parquet files.


These input and output formats and SerDes functions are fundamental to how Hive operates. They are used by Hive to:

Extract records: The InputFormat determines how the raw data in HDFS is split into individual records for processing. For instance, TextInputFormat treats each line as a record.

Extract fields from records: Once a record is identified, the SerDe (specifically, the Deserializer part) takes that record (a sequence of bytes) and parses it into individual fields, assigning them to the appropriate columns based on the table's schema defined in the Metastore. This includes handling data type conversions (e.g., converting a string "123" to an integer 123).

This comprehensive metadata stored in the Metastore is what allows Hive to provide its "schema on read" capability and translate SQL queries into low-level MapReduce (or other execution engine) operations that correctly interpret and process the diverse data formats stored in HDFS. It's the brain of Hive, enabling it to bridge the gap between structured SQL and unstructured or semi-structured data in a distributed file system.

Hive provides a Command Line Interface (CLI) that accepts and parses HQL commands. This traditional interface allows users to directly type and execute HQL queries, view results, and manage Hive objects. While functional, for more modern workflows, other interfaces are often preferred.


Hive also provides JDBC/ODBC connectors (drivers) to facilitate integration with a wide array of other tools. This broad connectivity is a major reason for Hive's popularity, as it allows users to leverage their existing skill sets and preferred applications for data analysis and visualization. These tools include:

beeline (CLI): A more modern and robust command-line client for Hive, offering better security features and connection management than the legacy Hive CLI. It connects via JDBC.

Oracle SQL Developer (GUI): A popular graphical user interface for database development and administration. With the appropriate JDBC driver, analysts can use SQL Developer to connect to Hive, browse schemas, and execute HQL queries in a familiar environment.

Talend Open Studio (data extraction, transformation, loading, and integration tools): An open-source data integration platform. Talend can use Hive's JDBC connectivity to extract data from Hive, perform transformations, and load it into other systems, or vice versa, making it a key component in data pipelines.

JasperReports, QlikView (business intelligence reporting tools): These BI tools can connect to Hive via JDBC/ODBC to pull data for generating reports, dashboards, and interactive visualizations, enabling business users to gain insights from their big data.

Microsoft Excel 2013 (data analysis tools): Excel's Power Query (or Get & Transform Data) feature can use the ODBC driver to connect to Hive, allowing users to import data directly into spreadsheets for further analysis, pivot tables, and charting, bringing big data closer to everyday business users.

Tableau (data visualization tools): A leading data visualization platform. Tableau offers a native Hive connector (via JDBC/ODBC) that allows users to create highly interactive and compelling visualizations from data stored in Hive, enabling deep data exploration.


HUE (Hadoop User Experience) provides a unified web interface to HDFS and Hive in an interactive environment. HUE simplifies interaction with the Hadoop ecosystem by offering a browser-based portal where users can:

Browse HDFS files.

Write and execute HQL queries against Hive.

View query results and execution logs.

Manage Hive tables and databases.

Interact with other Hadoop services like Pig, Oozie, and YARN.

HUE significantly improves the user experience for working with Hadoop.

Finally, HCatalog provides a metadata management system for Hadoop, Pig, Hive, and MapReduce. HCatalog is built on top of Hive's Metastore and exposes the metadata through a set of APIs, allowing various Hadoop components to share and understand the schema of data stored in HDFS.

HQL (Hive Query Language) is a powerful SQL-like language that provides a comprehensive set of commands for interacting with data in Hive. It can be broadly categorized into several functional areas:

Data Definition Language (DDL): This part of HQL is used for creating, deleting, and altering schema objects. These objects define the structure and organization of your data within Hive.

Data Selection and Scope Language: This is primarily concerned with querying data, linking data from different sources, and limiting the scope of the data retrieved.

Data Manipulation Language (DML): This part of HQL focuses on changing the data itself, including operations like inserting, updating, and deleting. While Hive was traditionally more focused on batch inserts, modern Hive versions have enhanced DML capabilities.

Data Aggregation and Sampling Language: This category of HQL is used for summarizing, grouping, and analyzing data, as well as for selecting subsets of data.

Sampling: Hive provides mechanisms for sampling data, which is useful for quick analysis on large datasets without processing the entire dataset.


In summary, HQL offers a rich set of commands that allow users to define their data structures, query and filter massive datasets, manipulate data, and perform complex aggregations and analyses, all within the familiar paradigm of SQL.

  • Data Manipulation Language is used for exchanging, moving, sorting, and transforming data

Similarities:

Despite their underlying architectural differences, Hive intentionally mimics many aspects of RDBMS to provide a familiar experience for data analysts and developers:

Tabular view of data objects: Both Hive and RDBMS present data in a structured, tabular format, organized into rows and columns. This abstraction is fundamental to how users interact with the data.

Directories and files viewed as tables: In Hive, the distributed files and directories in HDFS are logically mapped onto tables. Users query these "tables" as if they were conventional database tables, abstracting away the complexities of distributed file storage.

Types of columns in tables: Both systems enforce data types for columns (e.g., INT, STRING, BOOLEAN, FLOAT, DOUBLE, TIMESTAMP, etc.), ensuring data integrity at a schema level.

ccess to tables through HQL very similar to SQL: HQL's syntax is heavily inspired by SQL-92, making it easy for users with SQL experience to adapt to Hive. Commands like SELECT, FROM, WHERE, GROUP BY, JOIN, CREATE TABLE, INSERT INTO, and ALTER TABLE are largely consistent.

API interface the same as JDBC programming interface: Hive provides a JDBC (Java Database Connectivity) driver, allowing applications to connect and interact with Hive using standard JDBC APIs. This enables broad integration with existing tools and programming paradigms. Many BI tools and ETL platforms use this JDBC interface to connect to Hive, just as they would to an RDBMS.


Differences:

The core architectural differences between Hive and RDBMS stem from their primary design goals and the environments they operate in:

Load and read-only data management system based on implementation of HDFS:

Hive: Primarily designed for batch processing and analytical queries on large volumes of historical data. It's optimized for high-throughput reads and appends, often performing full table scans. Data loading typically involves adding new files to HDFS directories that correspond to tables. While modern Hive with ACID transactions supports updates and deletes, its fundamental design still leans towards append-only and bulk operations on immutable data.

RDBMS: Designed for online transaction processing (OLTP), handling frequent, small, and fast read/write/update operations. They are optimized for transactional integrity and quick lookup of individual records.

Hive: Historically, UPDATE operations were not supported or were implemented as a "coarse-grained transformation." This meant that an UPDATE on a few rows might effectively involve re-writing entire partitions or even the entire table. With the introduction of ACID transactions (via ORC files and the transactional property), Hive now offers more "fine-grained" updates and deletes, but it's still distinct from how RDBMS handle them. The underlying mechanism often involves writing new delta files and compaction processes.

RDBMS: UPDATE operations are inherently "fine-grained." Changes are applied directly to specific rows or blocks on disk with atomicity and isolation, making them efficient for single-record modifications.

No transaction processing system:

Hive: Historically, Hive lacked a traditional ACID (Atomicity, Consistency, Isolation, Durability) transaction processing system. Operations were not atomic across multiple statements, and concurrent writes could lead to inconsistencies. This was acceptable for its primary analytical workload where eventual consistency was often sufficient.

RDBMS: ACID compliance is a cornerstone of RDBMS design. Transactions ensure that database operations are reliable, even in the event of errors or concurrent access, critical for applications requiring high data integrity (e.g., banking systems).


Hive versus relational DBMS

  • Similarities
    • Tabular view of data objects in HDFS
    • Directories and files viewed as tables
    • Types of columns in tables
    • Access to tables through HQL very similar to SQL
    • API interface the same as JDBC programming interface
  • Differences
    • Load and read-only data management system based on implementation of HDFS
    • It is still possible to access data visible in tabular format in Hive directly through HDFS
    • UPDATE supported as coarse-grained transformation instead of fine-grained transformation in relational DBMSs
    • No transaction processing system
    • No verification of consistency constraints, e.g. primary keys, foreign keys, domains constraints, etc

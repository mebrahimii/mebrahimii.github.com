HBase is the open-source implementation of the concepts outlined in the seminal Google BigTable paper. BigTable is Google's proprietary highly available, high-performance distributed storage system.

As a distributed database, its data is spread across many commodity servers (a cluster), which allows it to handle petabytes of data and scale out horizontally.

HBase provides a BigTable view of data stored in HDFS:

HDFS (Hadoop Distributed File System) is the underlying storage layer for HBase. HBase doesn't store data directly on individual disks; it uses HDFS for persistent storage and fault tolerance.

HBase acts as a layer on top of HDFS, providing the structured, indexed, random-access capabilities that HDFS lacks (HDFS is primarily for large, sequential file access).

HBase is also called as Hadoop DataBase:

It's a common nickname because of its tight integration with the Apache Hadoop ecosystem (specifically HDFS and MapReduce), serving as the primary NoSQL database for large-scale data processing within that stack.


Data Model: The Sparse, Multidimensional Map

HBase still provides a tabular view of data however it is also very different from the traditional relational data model:

It's tabular in the sense that data is organized into rows and columns, like a spreadsheet.

However, it's a NoSQL (Not only SQL) database, fundamentally different from relational databases (like MySQL or PostgreSQL). It doesn't have fixed schemas, relationships (joins), or transactions in the traditional SQL sense, making it much more flexible and scalable for massive datasets.

HBase data model is a sparse, distributed, persistent multidimensional sorted map: This is the most formal and accurate definition:

Sparse: Tables can have a huge number of potential columns, but any given row only needs to use a small subset of them. Unused columns don't take up space, making it efficient for rows with highly variable data structures.

Distributed: The map (data) is partitioned and spread across the servers in the cluster (called RegionServers).

Persistent: Data is stored durably on HDFS.

Multidimensional Sorted Map: This defines how the data is uniquely addressed and stored:

The data structure is a map (key-value store) where the values are stored based on a composite key.

It is indexed by a row key, column key, and timestamp: These three components form the coordinates that uniquely identify any single data point (cell value) in HBase:

Row Key: A byte array that uniquely identifies a row. All rows are lexicographically sorted by the row key, which is critical for efficient range-based scanning and data locality.

Column Key (Column Family:Qualifier): A combination of a mandatory Column Family (defined at table creation, groups related columns, and determines storage properties) and a flexible Column Qualifier (user-defined name for the actual "column").

Timestamp: The time (version) at which the cell value was written. HBase keeps multiple versions of data, indexed by timestamp, allowing for time-based retrieval.

In essence, the unique address for any data value in HBase is: (Table, RowKey, ColumnFamily, ColumnQualifier, Timestamp) → CellValue.

Data Grouping: Column Families

Data within a row is grouped by a column family:

Column Families are the most critical structural and physical component. They are defined when the table is created and are fixed.

They serve as the physical storage unit. All data belonging to the same column family is stored together on HDFS in a separate set of files called HFiles.

Column families have an important impact on the physical implementation of HBase table:

This is the key difference from relational databases. Different column families can have different storage properties (e.g., compression, block size, number of versions to keep).

Because they are stored separately, accessing data from one column family does not require reading data from another, which greatly optimizes I/O performance.

Every row has the same column families although some of them can be empty:

While every row conceptually includes all defined column families, a row is sparse. If a row doesn't have data for a column family, it simply doesn't take up any physical space for that family, supporting the flexible NoSQL model.


Cell Identification and Data Typing

Data within a column family is addressed via its column qualifier, or simply, column name:

The Column Qualifier (or Column Name) is a user-defined tag (a byte array) that identifies a specific data field within a Column Family. Unlike the Column Family, qualifiers can be defined on the fly without needing a schema update.

Hence, a combination of row key, column family, and column qualifier uniquely identifies a cell:

This combination (along with the optional Timestamp) is the full address for a piece of data: (Row Key, Column Family:Qualifier) → Value.

Values in cells do not have a data type and are always treated as sequences of bytes:


HBase is type-agnostic. It simply stores and retrieves raw byte arrays.

It is the client application's responsibility to serialize (convert) application-level types (like integers, floats, strings, JSON objects) into bytes before writing and deserialize them back after reading. This keeps HBase fast and flexible.

Data Versioning in HBase

Values within a cell have multiple versions:

Unlike many traditional databases that only store the current value of a cell, HBase can retain multiple historical values (versions) for the same cell coordinate (Row Key, Column Family, and Column Qualifier).

This built-in versioning allows users to easily retrieve the value of a cell as it existed at a specific point in time, which is useful for audit logging, time-series data, or simply retrieving the previous state of data.


Versions are identified by their version number, which by default is a timestamp when the cell was written:

The timestamp is the default and most common way a version is defined. When a new value is written to a cell, HBase automatically assigns the current server time (in milliseconds) as the version number.

Crucially, when data is read, the versions are retrieved in descending order (most recent first). If a user doesn't specify a timestamp, they automatically get the latest version.


Customization: While the default is the millisecond timestamp, users can optionally specify their own custom version number when writing data, as long as it's a positive 64-bit integer, allowing for more controlled version tracking.


Summary of the Cell Address

Combining all the points previously discussed, the complete coordinate that uniquely identifies a specific piece of data (Value) in HBase is:

Address=(Row Key,Column Family,Column Qualifier,Timestamp)

This structure is what makes HBase a sparse, distributed, persistent multidimensional sorted map.

Timestamp and Version Defaults

If a timestamp is not determined at write time, then the current timestamp is used:

When an application writes data to an HBase cell and does not explicitly provide a timestamp, HBase automatically uses the current time on the server (in milliseconds since the Unix epoch) as the version identifier for that new value.

This ensures every write receives a unique, ordered version, which is the default and most common behavior.

If a timestamp is not determined during a read, the latest one is returned:

When an application reads a cell and does not specify a particular timestamp (version number) it's looking for, HBase, by default, returns only the single most recent version (the one with the highest timestamp).

This is the standard behavior for simple "get" operations, providing the "current" view of the data. You must explicitly request a historical version or request multiple versions during a read operation to see older data.


Version Control

The maximum allowed number of cell value versions is determined for each column family:

The Column Family is the administrative unit for versioning. When you create or alter an HBase table, you set the maximum number of versions (MAX_VERSIONS) that HBase should keep for every cell within that specific column family.

Once this limit is reached, the oldest version is automatically and eventually discarded (deleted) during compaction to conserve storage space.

The default number of cell versions is three:


f no specific MAX_VERSIONS parameter is set when defining a column family, HBase uses a default of three versions. This means it keeps the latest value and the two previous values for every cell in that family.

For applications requiring a full history (like time-series), this is often increased to a very large number, while for simple mutable data, it might be set to one.

Defining the Retrieval Key

The term "key" in an HBase read operation is flexible and acts as the address range or coordinate needed to isolate the desired data. It can range from a very broad row identifier to a very specific cell identifier.

Retrieval Goal Required Key Components HBase Operation Explanation

Entire Row Row Key Get (on the Row Key) If all the cells in a row are of interest, you only need the Row Key. HBase will return all Column Families, all Qualifiers, and, by default, the latest version of every cell within that specific row.

Specific Cells Row Key + Column Family and/or Qualifier Get (on Row Key + Scopes) If only specific cells are of interest, you must include the appropriate Column Families and/or Qualifiers along with the Row Key to narrow the scope.

Specific Version Row Key + CF + CQ + Timestamp Get (with Version Specified) To retrieve a historical value, you must add the specific Timestamp (version number) to the key components.


Granularity of the ReadHBase uses the provided components to target its underlying storage files (HFiles), making reads extremely efficient when you provide a precise address:

1. Broad Read: Row Key Only

When you provide just the Row Key, HBase identifies the RegionServer and Region holding that row. It then sequentially checks all the HFiles associated with every Column Family for that row.

Example: Retrieve everything about user "Row-0001".

Key: Row-0001

Result: You get the name, all phones, office phone, and address data.


2. Narrow Read: Row Key + Scope

When you add Column Families and/or Qualifiers, you significantly narrow the search scope:

Filtering by Column Family (CF): HBase only has to check the HFiles belonging to the specified CFs, ignoring all others. This provides a huge performance gain since CFs are physically separated.

Example: Retrieve only personal information for "Row-0001".

Key: Row-0001 + Column Family: Home

Filtering by Column Qualifier (CQ): You can narrow it further to a specific data field within a CF.

Example: Retrieve only the person's name.

Key: Row-0001 + Column Family: Home + Qualifier: Name

By allowing the user to construct a key with varying levels of detail, HBase supports flexible and highly performant access to its sparse, multidimensional data structure.

HBase Table Design Questions

Designing an HBase table is essentially making decisions that impact data distribution, read/write performance, and storage efficiency. Here's why those questions are crucial:


What should be a row key and what should it contain?

The Row Key is the most important design decision. It determines how data is sorted (lexicographically) and distributed across the cluster (sharded into Regions).

A good Row Key design ensures even data distribution (avoiding hotspots) and allows for efficient range scans (retrieving related rows quickly). For example, using reversed timestamps or salting can prevent hotspots when data is naturally sequential.


How many column families should a table have?

Column families are the physical storage unit. Data from different families are stored in separate HFiles.


Keep the number low (ideally 1–3). Too many CFs lead to a proliferation of files and inefficient resource management.

Grouping: Families should group columns that are often accessed together and have the same storage characteristics (e.g., compression, number of versions).

What columns (qualifiers) should be included in each column family?

The qualifiers are flexible and can be added on the fly. The main decision is to ensure that a qualifier belongs to the CF that best suits its access pattern and storage properties.


What information should go into the cells?

HBase is type-agnostic; cells only store raw bytes. The information should be structured (e.g., serialized into JSON, Avro, or protobuf) to minimize storage size and facilitate quick deserialization by the client application.


How many versions should be stored for each cell?

The maximum number of versions (MAX_VERSIONS) is set per Column Family.

This decision is a trade-off between data history retention (higher versions) and storage space/compaction time (lower versions). Set it to 1 if you only need the latest value, or to a high number for time-series data.

Data Access and Indexing

Indexing is performed only for a row key:

HBase is not a general-purpose relational database. It only provides an index on the Row Key.

This makes retrieval extremely fast when accessing a single row by its key or scanning a range of rows based on key prefixes.

Searching or querying data based on any column value (Qualifier) requires scanning large portions of the table (a full table scan), which is inherently slow. If secondary indexing is required, external tools like Apache Phoenix or ElasticSearch must be used.


HBase tables are stored sorted based on a row key:

All rows are stored in lexicographical order by the Row Key.

This is the basis of its scalability, as contiguous blocks of rows (Regions) can be stored on separate servers, enabling efficient range scans of related data.


Data Type and Atomicity

Everything in HBase tables is stored as untyped sequence of bytes:

This applies to Row Keys, Column Families, Column Qualifiers, and Cell Values. HBase is type-agnostic.

The client application is responsible for serializing (converting) complex data types (integers, strings, JSON objects) into bytes before writing and deserializing them back when reading.

Atomicity is guaranteed only at a row level and there are no multi-row transactions:

Atomicity means that all changes within a single row either succeed completely or fail completely. You will never see a partially updated row.

However, HBase does not support transactions that span multiple rows. This is a common trade-off in NoSQL databases to achieve massive horizontal scalability and high availability (sacrificing strict ACID compliance for a single-server system).


Column Family vs. Qualifier

Column families must be defined at HBase table creation time:

Column Families are the physical organizational unit and are part of the table's fixed schema. They determine how data is stored on HDFS (separate files) and managed (compression, versions).

They cannot be added or changed dynamically without altering the table schema.

Column qualifiers are dynamic and can be defined at write time:

Column Qualifiers are the logical columns within a family and are schemaless. You can define a new qualifier simply by writing data to it.

This allows for the storage of sparse and highly flexible data structures within the strict physical framework of the Column Families.

Column qualifiers are stored as sequences of bytes such that they can represent data:

Since qualifiers are bytes, they can also store variable data. For instance, in time-series data, the qualifier might represent a device ID or a specific metric name, making the table highly dynamic.

    • Column families must be defined at Hbase table creation time
    • Column qualifiers are dynamic and can be defined at write time

This HBase structure perfectly illustrates the implementation of a Fact Table with Denormalized Dimensions, a common pattern in data warehousing known as a star schema or denormalization, adapted for HBase.


Fact with Denormalized Dimensions

In data warehousing, a Fact is a core event or transaction (like a sale or a measurement), and Dimensions are the descriptive attributes related to that event (like the buyer, the seller, or the time).

Here's how the HBase design maps to this concept:


1. The Fact Record

Row Key: "1234567"

The Row Key represents the unique identifier for the Fact—the actual transaction or event that occurred. It's the primary key of the transaction record itself.

Fact Measures (Column Family: MEASURE)

This Column Family stores the quantifiable metric of the fact.

Qualifier: "amount" (the fact measure)

Value: "1000000"

This is the central data point, the "what" of the transaction.


2. The Denormalized Dimensions

Instead of storing only a foreign key (like a buyer_id) and requiring a separate lookup to get the buyer's details (which would be slow in HBase), the related dimensional data is denormalized and stored directly in the same row using dedicated Column Families.

Dimension 1: Buyer (Column Family: BUYER)

This CF stores the descriptive details of the person initiating the transaction.

Qualifiers: "phone", "first-name", "last-name".

Advantage: When you retrieve the fact "1234567", you get the buyer's full contact information immediately, avoiding a second query.

Dimension 2: Seller (Column Family: SELLER)

This CF stores the descriptive details of the person completing the transaction.

Qualifiers: "phone", "first-name", "last-name".

This structure demonstrates a method for implementing a graph database structure using HBase, where the graph is modeled as a collection of Nodes (entities) and Edges (relationships). This is achieved by leveraging the Row Key and Column Qualifiers to represent the graph components.


HBase Implementation of a Graph Structure

In this design, HBase is used to store the graph's adjacency list, defining the neighbors (nodes) connected to a central node via specific relationships (edges).

1. Nodes (Entities)

Row Key: The Row Key is used to uniquely identify a Node (or Vertex) in the graph.

Examples: "A", "B". These are the entities whose outgoing connections are being listed.


2. Edge Labels (Relationship Types)

Column Family (CF): The Column Family is used to represent the Edge Label or the Type of Relationship that connects the central node to its neighbors.

Examples: "R", "S". The relationships originating from Node "A" are categorized as type "R" and type "S".


3. Edges (The Link)

Column Qualifier (CQ): The Column Qualifier acts as the Edge ID or a unique identifier for a specific instance of the relationship.

Examples: "1", "2". These differentiate individual edges of the same type (CF) originating from the same node (Row Key).


4. Target Nodes (The Neighbors)

Cell Value: The Cell Value stores the Row Key (ID) of the Target Node (the neighbor) reached via the Edge.


Examples: "B", "D", "C". These are the nodes connected to the Row Key node.

HBase Architecture: Built on HDFS

HBase is a database built on top of HDFS:

HDFS (Hadoop Distributed File System) provides the persistence and fault tolerance for HBase. HBase does not manage its own files; it delegates this job to HDFS.

HBase acts as the random-access layer, providing features like indexing, versioning, and atomic row updates that HDFS (a sequential file system) lacks.

HBase tables can scale up to billions of rows and millions of columns:

This is the goal of the architecture. By horizontally scaling, HBase can handle datasets that grow to terabytes or even petabytes in size, far beyond the capacity of a single traditional database server.


Data Partitioning: Regions and RegionServers

HBase tables are split into smaller chunks of data that are distributed across multiple servers:

This partitioning is the mechanism for distributing the data and workload, enabling parallel processing and high scalability.

Chunks of data are called as regions and servers that host regions are called as region servers:

A Region is the fundamental unit of scaling and distribution. It represents a contiguous, sorted range of rows from a single table.

A RegionServer is a Java application that runs on a commodity server. Its job is to host and manage several Regions, handle read and write requests for the data stored in them, and communicate with HDFS.

Region servers are usually collocated with data nodes of HDFS:

This is a critical best practice called data locality. By running the RegionServer (which processes the data) on the same physical machine as the HDFS DataNode (which stores the data), HBase minimizes network traffic. When a RegionServer needs data, it can access it locally without having to fetch it across the network.


Types of Data Splits

The splits of HBase tables are usually horizontal, however, it is also possible to benefit from vertical splits separating column families:

Horizontal Split (Regions): This is the primary split. The table is divided by its Row Key range (e.g., Row Keys A-M go to Region 1, N-Z go to Region 2). This is how data is distributed across different RegionServers.

Vertical Split (Column Families): While not a physical split across servers, data within a single Region is separated vertically by Column Family (CF). All data belonging to one CF is stored in a separate set of files (HFiles) from another CF. This separation is crucial for I/O efficiency, as querying one CF does not require reading the files of other CFs.


Dynamic Management

Region assignments happen when HBase table grows in size or when a region server is malfunctioning or when a new region server is added:

Automatic Splitting: When a Region accumulates too much data (exceeds a configured size limit), it automatically splits into two new Regions. The HBase Master Server then reassigns one of the new Regions to a different RegionServer to balance the load.

Load Balancing and Recovery: If a RegionServer fails, its Regions are quickly reassigned to other active RegionServers. Likewise, adding a new RegionServer causes the Master to reassign existing Regions to the new server to improve cluster utilization. This ensures fault tolerance and elastic scalability.

The combination of HDFS for durability, Regions for horizontal scalability, and Column Families for vertical I/O efficiency is what allows HBase to operate as a high-performance, petabyte-scale database.

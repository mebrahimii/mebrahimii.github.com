What is Pig?

Pig is a high-level platform designed to make it easier to write programs for analyzing large datasets using Hadoop. It provides an engine for processing data flows in parallel on Hadoop.

Key Components and Functionality


Pig Latin Language:

It includes a scripting language called Pig Latin, which is used to represent the data flows.

Pig Latin offers operators for many traditional data operations like join, sort, filter, group, and load/store, allowing users to specify complex data transformations.

A key feature is the ability for users to develop their own custom functions (User Defined Functions or UDFs) for reading, processing, and writing data, extending Pig's capabilities.


Relationship with Hadoop:

Pig is built to run on the Hadoop ecosystem.

It utilizes the Hadoop Distributed File System (HDFS) for storage.

Critically, it is built on top of the Hadoop processing framework, MapReduce. Pig Latin scripts are automatically translated into a series of MapReduce jobs that run on the Hadoop cluster, handling the complex parallel programming details for the user.


In essence, Pig simplifies the process of working with Hadoop's MapReduce by providing a more accessible, relational-database-like scripting language (Pig Latin) for data analysis.


What is Pig ?

  • Pig provides an engine for processing data flows in parallel on Hadoop
  • It includes a language Pig Latin, for representation of the data flows
  • Pig Latin includes operators for many of the traditional data operations (join, sort, filter, etc.), as well as the ability for users to develop their own functions for reading, processing, and writing data
  • Pig runs on Hadoop, in the sense that it makes use of both HDFS, and is built on top of Hadoop processing framework MapReduce

Pig Latin: A Dataflow Language

Pig Latin is the language used within the Apache Pig platform. It is defined as a dataflow language, which means its primary purpose is to let users describe the flow of data through a series of transformations.

Describing the Data Flow


Focus on Movement and Transformation: Pig Latin allows users to specify how data from one or more inputs should be read, processed, and then stored to one or more outputs.


Parallel Execution: A crucial aspect is that Pig processes this data flow in parallel across a Hadoop cluster, which is essential for handling big data.


Structure (DAG): The entire sequence of data transformations in Pig is internally represented by a Directed Acyclic Graph (DAG). This is a common structure in data processing that ensures operations happen in a specific, non-looping order.


Absence of Control Flow: Unlike general-purpose programming languages, Pig Latin does not include traditional control-flow constructs like if statements or for loops. It focuses strictly on the data manipulations.


Pig Latin vs. SQL

The text highlights a key difference when comparing the dataflow language Pig Latin to a traditional query language like SQL:

Feature Pig Latin (Dataflow) SQL (Query)

Goal Specifies how data are manipulated through a sequence of operations. Oriented around answering one question (a single query).

Design Designed for a sequence of operations (the data pipeline). Designed to execute a single, specific query.


In short, SQL is about what result you want from the data right now, while Pig Latin is about building a pipeline or recipe that dictates how the raw input data is read, transformed step-by-step, and eventually written out as processed output.


Data flow language

  • The language of Pig, called Pig Latin is a dataflow language
  • It allows the users to describe how data from one or more inputs should be read, processed, and then stored to one or more outputs in parallel
  • The data flow in Pig is represented by a Directed Acyclic Graph (DAG)
  • There is no if statement or for loops in the Pig Latin
  • Comparing query and data flow languages:
    • SQL is oriented around answering one question, while Pig specifies how data are manipulated
    • SQL is oriented around answering one query, while Pig is designed for sequences of operations

WordCount Example in Pig Latin

The goal of this script is to read a file, count the frequency of every word within it, and print the results.

1. Load the Data

Pig Latin Command Explanation

input = load 'file' as (line); This command loads the data from the specified 'file' (which, in the example, is named "Mary" in the comments). The entire contents of each line in the file are loaded as a single field named line within a relation called input.


2. Tokenize and Flatten

Pig Latin Command Explanation

words = foreach input generate flatten(TOKENIZE(line)) as word; This is the core transformation step:

TOKENIZE(line) This function splits the string stored in the line field into a collection of individual words (tokens).

flatten(...) Since TOKENIZE returns a collection of words within a single record, flatten is used to expand this collection. It takes the group of records and produces a separate record for each word.

as word The resulting single field in this new relation, words, is named word.

Result: The words relation is a list where each record contains a single field, word, holding one word from the original file.


3. Group the Words

Pig Latin Command Explanation

grpd = group words by word; This groups all the records in the words relation that have the same value in the word field.

Result: The grpd relation contains one record for every unique word. Each record has two fields: group (the unique word itself) and a bag named words (which holds all the original records that had that word).


4. Count and Output

Pig Latin Command Explanation

cntd = foreach grpd generate group, COUNT(words); This command generates the final output for each group:

group This is the unique word.

COUNT(words) This function counts the number of records (words) inside the words bag for that group. This gives the frequency of the word.

dump cntd; This command executes the entire script and prints the final relation, cntd, which contains the unique words and their corresponding counts.


Pig Latin

  • WordCount example in Pig Latin 
      input = load 'file' as (line);
-- Load input from the file named Mary, and call the single field in the record
   'line'

-- TOKENIZE splits the line into a field for each word
-- flatten takes the collection of records returned by TOKENIZE and produce
   a separate record for each one, calling the single field in the record word

grpd = group words by word;
-- Now group them together by each word

cntd = foreach grpd generate group, COUNT(words);
-- Count them

-- Print out the results

Pig vs. MapReduce: Key Differences

The primary distinction is their level of abstraction and how they are used for data processing on systems like Hadoop.

1. Level of Abstraction and Ease of Use

Feature Pig (Pig Latin) MapReduce (Java/Python)

Abstraction Level High-level linguistic abstraction. Low-level programming model.

Expressiveness Very expressive for common data tasks. Expressive, but requires more code for simple tasks.

Core Operations Operations like JOIN, FILTER, and GROUP BY are primitive (built-in commands). Implementing these standard operations is non-trivial and requires writing custom code for the map and reduce functions.

Development Time Significantly reduces development time. Development is more time-consuming.

Maintenance Cost Lower cost to write and maintain than custom Java/Python code. Higher cost to write and maintain.


2. Performance and Execution

Feature Pig MapReduce

Execution Pig Latin scripts are compiled and optimized into a sequence of MapReduce jobs for execution. Executed directly as a Java or Python program using the MapReduce framework.

Efficiency The execution of a Java program for MapReduce is usually more efficient than an equivalent Pig script, as the custom-written code can be highly optimized for the specific task. Generally more efficient due to direct control over the underlying logic.


3. Programming Paradigm

Feature Pig MapReduce

Programming Style Declarative/Procedural (data-flow language). Does not support object-oriented development.

Object Orientation Procedural/Functional (focused on map and reduce functions). Fully supports object-oriented development (e.g., in Java).


Summary

Pig (using Pig Latin) acts as a productivity layer on top of MapReduce. It lets data analysts quickly write complex data transformation logic using simple, SQL-like commands. It sacrifices a small amount of execution efficiency and object-oriented support for a massive increase in development speed and ease of maintenance.

Choose Pig for rapid prototyping, ETL (Extract, Transform, Load) processes, and data flows that rely heavily on standard operations (join, group, filter).

Choose MapReduce (direct Java/Python) when execution efficiency is critical or when the task requires highly custom, non-standard processing logic that would be difficult to express in Pig Latin.


Differences between Pig and MapReduce

  • MapReduce programming model is expressive but relatively low-level, while Pig can be seen as a linguistic abstraction on MapReduce
    • For example, the implementation of standard data-processing operations (such as join, filter, group by, etc.) is non-trivial in MapReduce, but these operations are primitive in Pig Latin
  • Pig Latin reduces the development time of MapReduce programming and is much lower cost to write and maintain than Java code for MapReduce
  • On the other hand, Pig does not support the object-oriented development
  • Also, the execution of Java programs for MapReduce is usually more efficient than an equivalent Pig script

Short History of Pig

Origin: Pig started as a research project at Yahoo! Research.

Motivation (The Problem):

Researchers found the MapReduce paradigm in Hadoop to be too low-level and rigid.

This led to a lot of custom user code that was difficult to maintain and reuse.

They also observed that many MapReduce users were uncomfortable with declarative languages like SQL.


The Solution (Pig Latin):

To address this, they designed a new language called Pig Latin.

Pig Latin was intended to be a "sweet spot" between the declarative style of SQL and the low-level, procedural style of MapReduce.


Development & Adoption:

Yahoo! Hadoop users began adopting the research prototype.

A development team was formed to turn it into a production-quality product.


Key Milestones:

The project was explained in a paper presented at SIGMOD in 2008.

The first Pig release came in September 2008.

Later that same year (2008), Pig graduated from the Incubator and became a subproject of Apache Hadoop.


A short history of Pig

  • Pig started out as a research project in Yahoo! Research, where Yahoo! scientists designed it and produced an initial implementation
  • As explained in a paper presented at SIGMOD in 2008, the researchers felt that the MapReduce paradigm presented by Hadoop is too low-level and rigid, and leads to a great deal of custom user code that is hard to maintain and reuse
  • At the same time they observed that many MapReduce users were not comfortable with declarative languages such as SQL
  • Thus they set out to produce a new language called Pig Latin that we have designed to fit in a sweet spot between the declarative style of SQL, and the low-level, procedural style of MapReduce
  • Yahoo! Hadoop users started to adopt Pig and a team of development engineers was assembled to take the research prototype and build it into a production-quality product
  • About this same time, in fall 2007, the first Pig release came a year later in September 2008. Later that same year, Pig graduated from the Incubator and became a subproject of Apache Hadoop

Grunt: The Interactive Pig Shell

Grunt is the interactive shell that allows users to execute Pig Latin statements and interact with the file system.

Functionality:

It enables users to enter Pig Latin interactively (statement by statement).

It provides a shell for users to interact with HDFS (Hadoop Distributed File System).


How to Enter Grunt:

To start the shell, simply type the command:

pig

This will launch the shell and display the prompt:

grunt >


Default Configuration:

When launched with just pig, it interacts with the cluster configuration (typically HDFS) that is set in the classpath of Pig.


Local Mode:

If you want the shell to interact with a local file system instead of the cluster, you use the local execution mode command:

pig -x local

Executing Pig Latin in Grunt

The Grunt shell allows you to enter Pig Latin commands one at a time.

Entering Commands: Commands are typed directly after the grunt> prompt.

pig

grunt> dividends = load "NYSE_dividends" as (exchange, symbol, data,dividend)

grunt> symbols = foreach dividends generate symbol;


Error Checking: Grunt performs basic syntax and semantic checking immediately upon entry. This helps catch simple errors quickly, even before the job is executed.

Example of Error: In the provided example, typing symbl instead of the correct alias symbol resulted in an error:

grunt> symbols = foreach dividends generate symbl;... Error during parsing. Invalid alias: symbl ...


Delayed Execution (Lazy Evaluation): Pig Latin commands are not executed immediately when you press Enter. Pig builds up a logical and physical plan of execution.

Execution only starts when a command that requires an output is encountered, specifically either a store or a dump command. The dump command is used to display the results to the screen.

grunt> dump symbols;


Quitting Grunt: To exit the interactive shell, use the quit command:

grunt> quit


Entering Pig Latin scripts in Grunt

  • Pig Latin commands can be entered directly into Grunt. For example: 
      
pig
grunt> dividends = load "NYSE_dividends" as (exchange, symbol, data,dividend)
grunt> symbols = foreach dividends generate symbl;
... Error during parsing. Invalid alias: symbl ...
grunt> symbols = foreach dividends generate symbol;

  • Pig will not start executing the Pig Latin you enter until it sees either a store or dump
  • However, it will do basic syntax and semantic checking to help you catch errors quickly
  • To quit Grunt, type

Complex Types: Map

The map is a structure that allows you to store key-value pairs, similar to a dictionary or hash table in other programming languages.

Structure: A map is a chararray (key) to data element (value) mapping.

Key and Value:

The key is always a chararray (string) and acts as an index to find the element.

The value can be any Pig type, including another complex type (like a map, tuple, or bag).

Type Flexibility: By default, there is no requirement that all values within a single map must be of the same type (i.e., it can be heterogeneous).

Map Constants: Maps are defined using a specific syntax:

Delimiters: Square brackets ([]) enclose the map.

Key-Value Separator: The hash symbol (#) separates the key and its value.

Pair Separator: A comma (,) separates key-value pairs.


Example:

The constant 'name'#'bob', 'age'#55] creates a map where:

Key 1: 'name' (chararray), Value 1: 'bob' (chararray).

Key 2: 'age' (chararray), Value 2: 55 (int).

This demonstrates the type heterogeneity, as one value is a chararray and the other is an int.


Complex Types

  • map
  • A map in Pig is a chararray to data element mapping, where that element can be any Pig type, including a complex type.
  • The chararray is called a key and is used as an index to find the element, referred to as the value
  • By default there is no requirement that all values in a map must be of the same type
  • map constants are formed using brackets to delimit the map, a hash between keys and values, and a comma between key-value pairs
  • For example, ['name'#'bob', 'age'#55] will create a map with two keys, "name" and "age"
  • Note that the first value is a chararray, and the second is an int

Complex Types: The Tuple

A tuple is one of the fundamental complex data types in Apache Pig (which is often used for analyzing large datasets stored in systems like Hadoop).


Key Characteristics of a Tuple

Ordered Collection: The elements within a tuple are kept in a specific sequence.

Fixed-Length: Once created, a tuple has a defined number of elements (fields).

Divided into Fields: A tuple is broken down into individual elements called fields. Each field holds one piece of data.


Heterogeneous Data Types: The data types of the fields within a single tuple do not have to be the same. For example, one field can be a string, and the next can be an integer.

Analogy to SQL: It's often compared to a row in a relational database (SQL), where the fields within the tuple are analogous to the columns of that row.

Syntax: Tuple constants are enclosed in parentheses () and the fields are separated by commas ,.


Example

The example provided, ('bob', 55), represents a single tuple.

Field Index (Position) Field Value Data Type SQL Analogy

0 'bob' chararray (String) Name (Column)

1 55 int (Integer) Age (Column)


Why Use Complex Types?

In Pig, complex types like tuples, bags, and maps are essential because they allow you to represent the hierarchical and structured data that is common in big data processing, moving beyond simple scalar types (like single numbers or strings). They enable the efficient grouping and manipulation of related data items.


Complex Types

  • tuple
  • A tuple is a fixed-length, ordered collection of Pig data elements
  • tuples are divided into fields, with each field containing one data element
  • These elements can be of any type and they do not all need to be the same type
  • A tuple is analogous to a row in SQL, with the fields being SQL columns
  • Tuple constants use parentheses to indicate the tuple and commas to delimit fields in the tuple
  • For example, ('bob', 55) describes a tuple constant with two fields

Complex Types: The Bag

A bag is a fundamental complex data structure in Pig designed to hold collections of related data records (tuples). It's most commonly used to represent a collection of rows that result from an operation like a GROUP BY.


Key Characteristics of a Bag

Collection of Tuples: The bag's contents are composed entirely of tuples.

Unordered: This is the most critical feature. The tuples within a bag have no guaranteed sequence or order.

Due to this lack of order, you cannot reference tuples by position (index), unlike elements in an array or list.

Variable Schema (Often): While a bag can have a schema, it's a flexible structure.


Schema Definition: If a schema is defined for the bag, it applies to all tuples contained within it. This means all tuples in a single bag are generally expected to have the same structure (same number of fields and data types).

Analogy: A bag is similar to a table in SQL, but specifically one where the order of rows is irrelevant (like a heap table).

Syntax: Bag constants are enclosed in braces {}, with the constituent tuples separated by commas.


Example

The example {('bob', 55), ('sally', 52), ('john', 25)} is a single bag containing three tuples.

Tuple 1 Tuple 2 Tuple 3

('bob', 55) ('sally', 52) ('john', 25)


Complex Types

  • bag
  • A bag is an unordered collection of tuples
  • Because it has no order, it is not possible to reference tuples in a bag by position
  • Like tuples, a bag can, but is not required to, have a schema associated with it
  • In the case of a bag, the schema describes all tuples within the bag
  • bag constants are constructed using braces, with tuples in the bag separated by commas
  • For example, {('bob', 55), ('sally', 52), ('john', 25)} constructs a bag with three tuples, each with two fields

Null Data in Pig

In Apache Pig, the concept of a NULL data element is inherited directly from SQL and relational database theory.


Key Characteristics

Applies to Any Type: Any data element in Pig, regardless of its type (scalar, tuple, bag, or map), can be NULL.

Meaning: Unknown Value: The fundamental definition of NULL in Pig (and SQL) is that the value is unknown. It doesn't mean the value is zero, an empty string, or an empty collection; it means we simply don't have the data for that field.


Pig/SQL NULL vs. Programming Language Nulls

This is the most important distinction when moving from programming languages (like C, Java, Python) to data processing environments (like Pig/SQL):

Feature Pig/SQL NULL Programming Language null (e.g., None in Python, null in Java)

Meaning Unknown Value. It represents missing or non-existent data. Missing Reference. It means the variable or reference points to no object or memory location.

Logic Requires three-valued logic (TRUE, FALSE, UNKNOWN). Comparisons involving NULL generally result in UNKNOWN. Typically part of standard Boolean logic (TRUE/FALSE).

Example Is 5 = NULL? ˇú Unknown Is my_object == null? ˇú True or False


Null Data

  • Pig includes the concept of a data element being null
  • Data of any type can be null
  • In Pig, the concept of null is the same as in SQL, which is completely different from the concept of null in C, Java, Python, etc
  • In Pig a null data element means the value is unknown

A schema in Pig is essentially the blueprint or structure of your data. It defines the names and data types of the fields within your tuples (records). Explicitly defining a schema is the best way to ensure Pig correctly interprets your data.

1. Explicit Schema Definition (With Data Types)

This is the preferred method as it provides Pig with the most complete information, allowing it to perform efficient type-checking and optimization.

Example

dividends = load 'NYSE_dividends' as

(exchange:chararray, symbol:chararray, date:chararray, dividend:float);

Key Points

You assign a field name (e.g., exchange) and a data type (e.g., :chararray) for every expected field.

The data is treated as having four fields with the specified types.


2. Explicit Schema Definition (Without Data Types)

You can still define a schema by only providing the field names.

Example

dividends = load 'NYSE_dividends' as (exchange, symbol, date, dividend);

Key Point

In this case, Pig assumes the data type for all fields is bytearray. A bytearray is a raw, untyped collection of bytes. While flexible, you'll need to explicitly cast or convert these fields later in your script to perform most calculations or comparisons.


Impact of Schema on Data Loading

When you define a schema, you are setting a strict expectation for the structure of your input data. Pig enforces this structure as the data is loaded:

Too Many Fields in the Data:

If the input record has more fields than specified in the schema, Pig will truncate the extra fields. Only the fields matching the schema definition are kept.

Too Few Fields in the Data:

If the input record has fewer fields than specified, Pig will pad the end of the record with NULLs for the missing fields.

For example, if the schema expects four fields but the data only has three, the fourth field will be set to NULL (unknown value).

Defining a schema is critical for data integrity and predictable processing in Pig Latin scripts.


Schemas

  • The easiest way to communicate the schema of your data to Pig is to explicitly tell Pig what it is when you load the data 
      
dividends = load 'NYSE_dividends' as
  (exchange:chararray, symbol:chararray, date:chararray, dividend:float);
  • Pig now expects your data to have four fields and if it has more, it will truncate the extra ones
  • If it has less, it will pad the end of the record with nulls
  • It is also possible to specify the schema without giving explicit data types. In this case, the data type is assumed to be bytearray 
      
dividends = load 'NYSE_dividends' as (exchange, symbol, date, dividend);

When you define a schema using the AS keyword in a LOAD statement, you can specify that a field should hold a complex data structure and even define the structure of that complex type.

1. Map (map)

A map is an associative array (key-value store). When declaring a map, you specify the data type of the values. The keys are always assumed to be chararray (string).

Syntax Option Example Description

map[] as (a:map[]) Declares a map where the data type of the values is unknown (assumed to be bytearray).

map[type] as (b:map[int]) Declares a map where the values are guaranteed to be of the specified type (e.g., int).


2. Tuple (tuple)A tuple is an ordered collection of fields. When declaring a tuple in a schema, you define the structure (fields) it contains.

Syntax Option Example Description

tuple() as (a:tuple()) Declares the field as a tuple, but the internal structure is unknown (assumed to be a tuple of bytearray fields).

tuple(list_of_fields) as (b:tuple(x:int, y:int)) Declares the field as a tuple and explicitly defines its schema: two fields named x (an integer) and y (an integer).


3. Bag (bag)

A bag is an unordered collection of tuples. When declaring a bag, you specify the schema of the tuples it contains.

Syntax Option Example Description

bag{} as (a:bag{}) Declares the field as a bag, but the internal tuple structure is unknown (assumes tuples of bytearray fields).

bag{t:(list_of_fields)} as (b:bag{t:(x:int, y:int)}) Declares the field as a bag and explicitly defines the schema of its tuples: a tuple named t containing fields x (int) and y (int).


Schemas

  • You can declare data as complex data types in a schema: 
      
Data type  Syntax                                                  Example
map        map[] or map[type], where type is any valid type        as (a:map[], b:map[int])
           This declares all values in the map to be of this type

tuple      tuple() or tuple(list_of_fields), where list_of_fields  as (a:tuple(), b:tuple(x:int, y:int))
           is a comma-separated list of field declarations

bag        tuple() or tuple(list_of_fields), where list_of_fields  as (a:bag{}, b:bag{t:(x:int, y:int)})
           is a comma-separated list of field declarations

Output of DESCRIBE dividends

When you define a schema without explicitly specifying data types, Pig assumes the raw, untyped type of bytearray for every field.

dividends = load 'NYSE_dividends' as (exchange, symbol, date, dividend);

described dividends;

Expected Output


The output of DESCRIBE dividends; would be:

dividends: {exchange: bytearray, symbol: bytearray, date: bytearray, dividend: bytearray}


Explanation

Field Names Preserved: Pig uses the descriptive names you provided (exchange, symbol, date, dividend).

Default Data Type: Because no specific type (like chararray, int, or float) was given, Pig defaults the type of the loaded data to bytearray. This is a placeholder for raw, untyped data (a stream of bytes).

Requirement for Processing: If you intend to perform mathematical operations, comparisons, or date functions on these fields, you would need to explicitly use the CAST operator later in your Pig script to convert the bytearray into the required type (e.g., float or int).


Schema

  • We can get the schema of a relation by using the describe operation 
      
dividends = load 'NYSE_dividends' as
            (exchange:chararray, symbol:chararray, date:chararray, dividend:float);
described dividends;
grpd: {exchange: chararray, stock: chararray, date: chararray,dividends: float}

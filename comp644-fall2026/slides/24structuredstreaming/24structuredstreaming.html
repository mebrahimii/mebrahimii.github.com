1. The "Unbounded Table" Concept

In traditional batch processing, you run a query against a static dataset (a table with a fixed number of rows). In Structured Streaming, Spark treats the incoming data stream as an Input Table that is infinitely long.

Batch: You process rows 1 through 1,000 and the job ends.

Streaming: You start at row 1, and as new data arrives at the source, it is simply "appended" to the end of the table. The table is unbounded because it never stops growing.

2. From Static Query to Incremental Execution

Because the stream is viewed as a table, you can write your logic using standard SQL or DataFrame APIs (like SELECT, GROUP BY, or JOIN).

The Query: You define a "Result Table" by applying a query to the Input Table.

The Trigger: At a set interval (e.g., every 1 second), Spark checks for new data.

Incremental Processing: Instead of re-reading the entire table, Spark’s engine identifies only the new rows (the "delta"), updates the result, and writes it to the output.

3. How Data Flows: Input to Output

The model follows a three-step cycle to ensure the "Result Table" stays up to date:

Component Description

Input Table The live stream where new data arrives as rows.

Query The logic (e.g., count()) you apply to the data.

Result Table The final state of your data after the query is applied.

Output What gets written to the external sink (a database, a file, or a console).

4. Output Modes

Since the Result Table is constantly changing, you have to decide how you want to see the results. Spark provides three main modes:

Complete Mode: The entire Result Table is written to storage every time. (Best for aggregations like "total count since the start").

Append Mode: Only the new rows added to the Result Table since the last trigger are written. (Best for simple transformations).

Update Mode: Only the rows that were changed in the Result Table since the last trigger are written.

Key Benefit: This model eliminates the need for the developer to worry about "streaming-specific" problems like event-time ordering, state management, or fault tolerance. Spark handles those under the hood so you can focus on the business logic.


Programming Model

  • The key idea in Structured Streaming is to treat a live data stream as a table that is being continuously appended
  • This leads to a new stream processing model that is very similar to a batch processing model
  • Users can express the streaming computation as standard batch-like query as on a static table, and Spark runs it as an incremental query on the unbounded Input Table
  • A new data item arriving on the stream is like a new row being appended to Input Table

Query The logic (e.g., SUM or COUNT) you apply to the data.

This model is powerful because it handles the hardest parts of streaming for you:

Fault Tolerance: If the system crashes, Spark uses "checkpointing" and "write-ahead logs" to know exactly which row it left off on.

Event-Time Processing: It can handle data that arrives late or out of order by looking at a timestamp column within the row rather than just when the data reached the server.

In Structured Streaming, the Input Source is the starting point of your pipeline. To treat a stream as a table, Spark needs a way to "read" the data continuously. This is handled by the DataStreamReader object.

Here is a breakdown of how to create these streams and the specific sources available:

1. The readStream() Interface

In standard Spark (Batch), you use spark.read. In Structured Streaming, you use spark.readStream(). This tells Spark to keep the connection open and watch for new data rather than doing a one-time pull.

# Basic syntax example

df = spark.readStream \

.format("source_type") \

.option("key", "value") \

.load()

2. Common Data Sources

Spark provides several built-in sources, ranging from testing tools to production-grade connectors:

File Source

Reads files written in a directory as a stream of data.

How it works: Spark monitors a folder (e.g., S3, HDFS, or local). When a new file is dropped into that folder, Spark treats the content of that file as new rows appended to the Input Table.

Supported formats: CSV, JSON, ORC, Parquet, Text.

Kafka Source

This is the most common production source. It polls data from Apache Kafka (a distributed messaging system).

Key Feature: It allows you to subscribe to one or more "topics." Spark handles the offsets (the position in the stream) automatically to ensure no data is lost if the system restarts.

Socket Source (Testing Only)

Reads UTF-8 text data from a network socket connection.

Use Case: Ideal for learning or quick debugging. You can use a tool like nc -lk 9999 to type words into a terminal and see Spark process them in real-time.

Limitation: It is not fault-tolerant and should never be used in production.

Rate Source

A "dummy" source used for testing and benchmarking.

How it works: It automatically generates data at a specific number of rows per second.

Structure: Each generated row contains two columns: timestamp and value (a sequential number). This is perfect for testing how your system handles a specific volume of traffic.

Rate Per Micro-Batch Source

Similar to the Rate source, but instead of rows-per-second, it generates a fixed number of rows per trigger (micro-batch).

Use Case: Useful for testing backpressure and seeing how your transformations behave when a specific "chunk" of data arrives all at once.


Input Source

  • Structured Streaming is based on creating streaming DataFrames
  • Streaming DataFrames can be created through the DataStreamReader interface returned by SparkSession.readStream()
  • Structured Streaming may have the followign data sources
    • File source: it reads files written in a directory as a stream of data
    • Kafka source: it reads data from Kafka
    • Socket source: it reads text data from a socket connection
    • Rate source: it generates data at the specified number of rows per second, each output row contains a timestamp and value
    • Rate Per Micro-Batch source: it generates data at the specified number of rows per micro-batch, each output row contains a timestamp and value

Apache Spark Structured Streaming, the Dataframe.writeStream() method is the entry point for turning a static DataFrame into a streaming query. Once you call .start(), the engine begins processing data continuously as it arrives.

Here is a breakdown of the key components you need to configure:

1. Output Sink

This defines where the data goes. You must specify the format (e.g., parquet, kafka, console, or memory) and often a path for file-based sinks.

Example: .format("parquet").option("path", "path/to/destination")

2. Output Mode

This defines what is written to the sink when new data arrives. There are three main types:

Append Mode: Only new rows added to the result table since the last trigger are written. (Default)

Complete Mode: The entire updated result table is written to the sink every time.

Update Mode: Only the rows that were updated since the last trigger are written.

3. Trigger Interval (Optional)

This defines the timing of the stream processing. If not specified, Spark will process data as soon as the previous micro-batch completes.

Fixed interval: e.g., .trigger(Trigger.ProcessingTime("10 seconds"))

One-time: .trigger(Trigger.Once()) processes all available data and then stops.

4. Checkpoint Location (Critical for Production)

This is a directory in an HDFS-compatible file system where Spark saves the state and progress of the query.

Why it matters: If the query fails, Spark uses the checkpoint to pick up exactly where it left off, ensuring "exactly-once" delivery guarantees.

Example: .option("checkpointLocation", "path/to/checkpoint/dir")

5. Query Name (Optional)

A unique string used to identify the query. This is particularly useful if you are running multiple queries in a single Spark application and want to monitor them via the Spark UI.

Basic Code Structure

query = df.writeStream \

.format("console") \

.outputMode("append") \

.queryName("my_streaming_query") \

.trigger(processingTime='5 seconds') \

.option("checkpointLocation", "/mnt/checkpoints/") \

.start()

query.awaitTermination()


Starting Stream Queries

  • To start the streaming computation we use Dataframe.writeStream() method
  • One or more of the following details must be provided
    • Details of the output sink: data format, location, etc.
    • Output mode: what gets written to the output sink
    • Query name: unique name of the query for identification (optional)
    • Trigger interval: trigger interval (optional)
    • Checkpoint location: location where the system will write all the checkpoint information (optional)

In Structured Streaming (like in Apache Spark), an Output Sink is the final destination for your processed data. Once the engine processes a "micro-batch" or a stream of data, it needs to know where to send the results.

Think of it as the exit point of a pipeline. Depending on your use case—whether you're archiving data, powering a real-time dashboard, or triggering another process—you choose a different sink.

1. File Sink

The File Sink is used to store the output of your stream into a permanent storage system (like HDFS, S3, or a local file system). It is ideal for long-term data storage and batch-processing later on.

Common Formats: Parquet (standard for performance), ORC, JSON, and CSV.

writeStream

.format("parquet")

.option("path", "path/to/destination/dir")

.option("checkpointLocation", "path/to/checkpoint/dir")

.start()

2. Kafka Sink

The Kafka Sink pushes your processed data into one or more Apache Kafka topics. This is the gold standard for "event-driven" architectures where one stream's output becomes another system's input.

Use Case: Sending alerts to a notification service or feeding data into a real-time analytics engine.

Requirements: You must provide the Kafka brokers (bootstrap servers) and the target topic name.

3. Foreach (and ForeachBatch) Sink

The Foreach Sink is the "Swiss Army Knife" of sinks. Use this when the built-in sinks (File, Kafka, etc.) don't meet your needs. It allows you to write custom logic for every row.

Use Case: Writing to a NoSQL database (like Cassandra or MongoDB), calling an external API for every record, or sending custom emails.

ForeachBatch: A more common variant that allows you to process an entire micro-batch at once using standard Batch Dataframe operations.

4. Console Sink

The Console Sink prints the output directly to your terminal or standard output.

Warning: This is strictly for debugging. Because it prints to the console, it cannot handle high volumes of data and does not provide "fault tolerance" (if it crashes, the data shown on screen is gone).

Modes: It supports Append (only show new rows) and Complete (show the entire result table every time).


Output Sinks

  • There are a few types of built-in output sinks
  • File sink: stores the output to a directory 
        
writeStream.format("parquet").option("path", "path/to/destination/dir").start()
	 // format can be "orc", "json", "csv", etc.
  • Kafka sink: stores the output to one or more topics in Kafka 
        
writeStream.format("kafka").option("kafka.bootstrap.servers", "host1:port1,host2:port2")
	                   .option("topic", "updates").start()
  • Foreach sink: runs arbitrary computation on the records in the output 
        
writeStream.foreach(...).start()
  • Console sink: prints the output to the console/stdoutfor every activation of a trigger, supports Append and Complete output modes 
        
writeStream.format("console").start()

What is a Memory Sink?

When you use the memory sink, Spark stores the output of your streaming query in the driver's memory as an in-memory table. This allows you to run standard SQL queries (or DataFrame operations) against the results in real-time.

Key Characteristics

Temporary Storage: The data is stored in the system memory (RAM). If the Spark driver crashes or the application stops, the data is lost.

Queryability: You must provide a queryName. This name becomes the string you use in the FROM clause of your SQL queries.

Not for Production: Because it stores all data in the driver's memory, it can easily cause OutOfMemory (OOM) errors with large datasets. It is strictly intended for low-volume testing.

Supported Output Modes

The memory sink supports two primary modes that define how data is written to the in-memory table:

Append Mode: Only the new rows added to the result table since the last trigger are added to the memory table.

Complete Mode: The entire result table is recalculated and replaces the previous version in memory every time there is new data. This is typically used for streaming aggregations (e.g., keeping a running count of words).

Syntax Breakdown

The code snippet you provided is the standard way to initialize this sink:

writeStream \

.format("memory") \

.queryName("tableName") \

.start()

.format("memory"): Tells Spark to use the memory sink provider.

.queryName("tableName"): Defines the name of the temporary table created in the Spark catalog.

.start(): Begins the execution of the streaming query.

How to view the results: Once the stream is running, you can view the data using a standard Spark SQL command:

spark.sql("SELECT * FROM tableName").show()


Output Sinks

  • Memory sink: outputs the results memory as an in-memory table, supports Append and Complete output modes 
        
writeStream.format("memory").queryName("tableName").start()

This completes the logic for the Structured Network Word Count. At this stage, you are transforming the raw input stream into a summarized result and defining how that result is delivered to you.

1. Splitting the Lines into Words

The input from the socket arrives as a single column named value (e.g., "hello world").

words = lines.select( explode( split(lines.value, " ") ).alias("word") )

explode(...): This is a powerful function that takes an array and turns every element into a new row.

2. Counting the Words

This is where the "Structured" part of Structured Streaming shines. You use the same syntax as a static DataFrame.

wordCounts = words.groupBy("word").count()

Spark maintains a state in the background. It remembers the counts of every word seen since the stream started and updates them as new data arrives.

3. Starting the Query (The Console Sink)

The query won't actually do anything until you call .start().

query = wordCounts.writeStream.outputMode("complete").format("console").start()

.outputMode("complete"): Because we are doing an aggregation (counting), "complete" mode ensures that every time the console updates, you see the entire result table (all words and their current total counts), not just the changes.

.format("console"): This tells Spark to print the results directly to your terminal or IDE log.

4. Awaiting Termination

Streaming jobs are designed to run forever.

query.awaitTermination()

This line tells the driver script to sit and wait while the background streaming process runs. Without this, the script would reach the end of the file and exit immediately, killing your stream before it even has a chance to process data.

from pyspark.sql import SparkSession

2. Data Type Transformation (Casting)

By default, the socket source reads everything as a String. To perform numerical comparisons (like >= 100), you must convert the data type.

In Structured Streaming, Window Operations allow you to perform aggregations (like counting words or averaging temperatures) over specific "chunks" of time. Instead of grouping by a category like deviceType, you group by a time interval.

1. The Concept of Event Time

Event Time is the time embedded within the data itself (e.g., a timestamp generated by an IoT sensor or a log entry), rather than the time the data reaches Spark. This is crucial because data often arrives out of order or delayed due to network lag.

2. Grouping by Windows

In a basic aggregation, a word like "apple" has one total count. In a windowed aggregation, "apple" has a different count for every time window in which it appeared.

The Example: 10-minute windows, updating every 5 minutes

This is known as a Sliding Window.

Window Duration (10 min): How long the window lasts.

Slide Duration (5 min): How often a new window starts.

Because the slide (5 min) is smaller than the duration (10 min), the windows overlap. This means a single event—like a word sent at 12:07—will actually fall into two different windows:

The 12:00 - 12:10 window.

The 12:05 - 12:15 window.

3. How the Result Table Looks

Spark represents these windows as a special column containing a struct with start and end times.

Window Word Count

12:00 - 12:10 hello 5

12:05 - 12:15 hello 3

12:10 - 12:20 hello 1

4. Why Use Sliding Windows?

Sliding windows are perfect for "moving averages" or detecting trends. If you only used Tumbling Windows (where windows don't overlap, e.g., 10:00-10:10 and 10:10-10:20), you might miss a spike in activity that happens exactly at 10:10. Sliding windows ensure that continuous activity is captured accurately.

In stream processing, Late Data refers to events that arrive at the system significantly after the timestamp recorded in the event itself. Because Spark Structured Streaming uses Event Time (as seen in your windowing example), it can intelligently handle these "stragglers."

1. Updating Past Aggregations

When late data arrives, Spark looks at the event’s timestamp, identifies which window(s) it belongs to, and re-calculates the aggregation for those specific windows.

State Store: Spark keeps the intermediate counts for previous windows in a "State Store."

Update: If an event from 12:03 arrives at 12:15, Spark finds the 12:00–12:10 window in the state store and increments the count.

Output: The system then emits an updated result for that window.

2. The Problem: Unbounded State

If Spark kept every window open forever to wait for late data, the system would eventually run out of memory. This is where Watermarking becomes essential.

A Watermark is a threshold that tells Spark how late data can be. It is defined as: Max Event Time seen by the engine - Late Threshold

3. How it works with Code

To handle late data efficiently, you chain the .withWatermark() operator before your grouping:

windowedCounts = words \

.withWatermark("timestamp", "10 minutes") \

.groupBy(

window(words.timestamp, "10 minutes", "5 minutes"),

words.word) \

.count()

The Lifecycle of Late Data:

Arrival: An event arrives with a timestamp older than the current time.

Validation: Spark checks the Watermark.

If the event is newer than the watermark, the state is updated.

If the event is older than the watermark, it is considered "too late" and is dropped (ignored) to save memory.

Cleanup: Once the watermark moves past the end of a window (plus the late threshold), Spark clears that window from its memory.

In stream processing, Watermarking is the mechanism used to balance two competing needs: the ability to handle late data and the need to free up system memory (State Store).

Without watermarking, Spark would have to keep the counts for every window in memory forever, because it would never know if a very old piece of data might eventually show up.

1. The Watermark Logic

The "Watermark" is a moving boundary calculated as:

Watermark=(Maximum event time seen so far)−(Allowed late threshold)

As Spark processes new data, it tracks the highest timestamp it has encountered. The watermark trails behind that maximum time by the threshold you define (e.g., 10 minutes).

2. The "Window Dropping" Rule

A window's state is maintained in memory only as long as the watermark hasn't passed the window's end time.

If Event Time > Watermark: The data is accepted, and the window aggregate is updated.

If Event Time < Watermark: The data is considered "too late" and is dropped.

Cleanup: Once the Watermark exceeds the End Time of a window, Spark assumes no more relevant data will arrive for that period and deletes that window from the state store to save memory.

3. Step-by-Step Example

Imagine we have a 10-minute window and a 5-minute watermark threshold.

Time Max Event Time Seen Calculated Watermark Action

12:15 12:15 12:10 (12:15 - 5m) Any data for the 12:00-12:10 window is still accepted.

12:20 12:20 12:15 (12:20 - 5m) The Watermark (12:15) is now greater than the end of the 12:00-12:10 window.

Result State Cleanup: The 12:00-12:10 window is cleared from memory.


Watermarking

  • Keeping old data for a very long time is not possible due to the transient memory limitations
  • Watermarking allows to track the current event time in the data and attempt to clean up old aggregations
  • A watermark of a query determines an event time column and the threshold on how late the data is expected to be in terms of event time
  • For example, if a window ends at time T, the system maintains state and allow late data to update the state until (max event time seen by the engine - late threshold > T)
  • Late data within the threshold are aggregated, but data later than the threshold are dropped

In Spark Structured Streaming, Watermarking combined with Update Mode is a powerful way to handle late data while keeping the system's memory usage under control.

1. How Watermarking Works here

The watermark is a moving threshold that lags behind the maximum event time seen by the engine. In your example:

Watermark=(Max Event Time seen)−10 minutes

Any data with a timestamp older than this calculated watermark is considered "too late" and is dropped. Any data newer than the watermark is accepted and used to update the counts.

2. Update Mode: The "Continuous Update" Logic

In Update Mode, the behavior is focused on providing the most recent results as soon as possible:

Immediate Output: As soon as a micro-batch is processed, any window that received new data (including late data) is updated in the Result Table and pushed to the sink.

Late Data Integration: If an event is late but still newer than the watermark, Spark retrieves the old window from its internal State Store, increments the count, and emits that updated row.

State Eviction (Cleanup): Once the watermark moves past the end of a specific window, Spark knows that no more "legal" late data can arrive for that window. It deletes that window from the State Store to free up memory. Once deleted, any future data for that window—no matter what—will be dropped.

1. The Core Behavior

In Update Mode, Spark outputs only the rows that were updated in the most recent micro-batch.

Low Latency: Unlike Append Mode (which waits for the watermark to pass before showing anything), Update Mode emits the "current best count" for a window as soon as the first event for that window arrives.

Continuous Refinement: If late data arrives later (but still within the watermark), Spark simply updates that row and sends it to the sink again.

2. How Watermarking Works in Update Mode

The watermark serves one primary purpose in Update Mode: State Cleanup.

Without a watermark, Spark would have to keep the counts for every window (e.g., 8:00 AM, 9:00 AM, 10:00 AM...) in its internal memory forever, just in case a stray event from 8:00 AM arrives tomorrow. This would eventually cause an Out of Memory (OOM) error.

The Cleanup Rule: A window is removed from the internal state store only when:

Watermark>Window End Time

Once a window is evicted from state, Spark "forgets" it entirely. If an event for that window arrives after eviction, it is dropped and no update is sent to the sink.

3. Comparison: Update Mode vs. Append Mode

Understanding the difference is critical for choosing the right sink (like a database or a console).

Feature Update Mode Append Mode

When is data emitted? Immediately (every trigger). Only after the watermark passes the window end.

Can rows be updated? Yes. The same window can be sent to the sink multiple times. No. Each window is emitted exactly once.

Memory Cleanup? When watermark > window end. When watermark > window end.

Best Use Case: Real-time dashboards (Grafana, PowerBI). Writing to files (Parquet/CSV) or non-updateable sinks.

In Append Mode, the behavior of Watermarking is fundamentally different from Update Mode. While Update Mode focuses on low latency (showing you results as they change), Append Mode focuses on finality and consistency.

1. The Core Rule of Append Mode

In Append Mode, Spark will only output a row to the sink when it is certain that the result for that window will never change again.

Since late data could arrive at any moment and change the count, Spark uses the Watermark as a "guarantee" of completion. A window is only finalized and "appended" to the result table once the Watermark has moved past the end of that window.

2. How the Process Works

Let’s use the example: 10-minute windows with a 10-minute watermark.

Data Arrives: Events with timestamps between 12:00 and 12:10 arrive.

Internal State: Spark calculates the counts internally but does not output anything to the sink yet.

Watermark Advances: The engine sees a new event with a timestamp of 12:20.

The Trigger: The Watermark is now 12:20−10 minutes=12:10.

Final Output: Because the Watermark (12:10) is now equal to (or greater than) the end of the 12:00–12:10 window, Spark:

Finalizes the count for that window.

Appends the result to the sink.

Drops the state from memory (cleanup).

3. Trade-offs: Latency vs. Stability

Append Mode creates a "lag" in your data equal to the window length plus the watermark threshold.

Feature Append Mode Detail

Output Latency High. You must wait for the "late threshold" to pass before seeing data.

Record Uniqueness Guaranteed. Each window/key pair appears in your output exactly once.

Sink Compatibility Perfect for Files (Parquet/JSON) or Append-only Databases where you cannot go back and "update" a row that was already written.

In Spark Structured Streaming, time windows are used to segment an infinite stream of data into finite chunks for aggregation. Depending on how you want to analyze your data, you can choose between three distinct types:

1. Tumbling Window (Fixed)

Tumbling windows are fixed-size, non-overlapping, and contiguous time intervals.

Behavior: Every event belongs to exactly one window.

When to use: When you need a simple summary of what happened in distinct blocks of time (e.g., "How many sales did we make every hour?").

Syntax: window(timestamp, "10 minutes")

2. Sliding Window

Sliding windows are also fixed-size, but they can overlap. They are defined by two parameters: the window duration (how long the window is) and the slide interval (how often a new window starts).

Behavior: If the slide interval is smaller than the duration, an event can belong to multiple windows.

When to use: When you need a "moving" average or a smoother trend (e.g., "What is the average CPU usage over the last 10 minutes, updated every 5 minutes?").

Syntax: window(timestamp, "10 minutes", "5 minutes")

3. Session Window

Session windows have dynamic sizes and are based on periods of activity. They do not have a fixed start or end time.

Behavior: A session window starts when an event arrives and stays open as long as new events keep arriving within a specified inactivity gap. If no data arrives for longer than that gap, the window closes.

When to use: When events are naturally grouped by user activity (e.g., "Track a user's website visit from the moment they land until they haven't clicked anything for 30 minutes").

Syntax: session_window(timestamp, "30 minutes")

In data processing and stream analytics (like Apache Flink, Spark, or Kafka Streams), Time Windows are used to bucket continuous data into manageable chunks for analysis.

Here is a breakdown of the three primary types of windows based on your description:

1. Tumbling Windows

Tumbling windows are the simplest form of windowing. They segment a data stream into fixed-size, non-overlapping, and contiguous time intervals.

Behavior: Once a window ends, a new one begins immediately.

Key Characteristic: Each data point belongs to exactly one window.

Use Case: Calculating total sales per hour where you don't want any data counted twice.

2. Sliding Windows

Sliding windows are also fixed-sized, but they add a "slide" parameter that defines how frequently a new window starts.

Behavior: If the slide duration is smaller than the window size, the windows overlap.

Key Characteristic: A single data point can be bound to multiple windows.

Use Case: Calculating a "moving average." For example, a 10-minute window that recalculates every 5 minutes.

3. Session Windows

Unlike the first two, Session windows do not have a fixed duration. Their size is dynamic and determined by the arrival of data.

Behavior: A window opens when an input arrives. It stays open as long as new data points arrive within a specified Gap Duration.

The "Gap": If no data is received for a period longer than the gap duration, the window closes.

Key Characteristic: They are ideal for activity that comes in "bursts" followed by periods of silence.

Use Case: Tracking user behavior on a website. The session stays active while the user clicks links but "times out" after 30 minutes of inactivity.

Building on your code snippet, it appears you are looking at Spark Structured Streaming syntax. Session windows are unique because they don't have a pre-defined start and end time; they are "data-driven."

Here is an explanation of how Session Windows function, specifically focusing on the dynamic gap features you mentioned.

1. The Session Window Mechanism

A session window groups events that occur close together in time. Unlike Tumbling or Sliding windows, these are defined by periods of activity separated by periods of inactivity (gaps).

The "Gap" Duration: In your code, "5 minutes" is the threshold. If a user triggers an event, the window stays open. If 5 minutes pass without another event from that same user, the session closes.

Watermarking: The .withWatermark("timestamp", "10 minutes") is crucial here. It tells Spark how long to wait for "late" data before finalizing the session and dropping it from memory.

2. Dynamic Gap Duration

The most advanced form of session windowing is when the gap is not a static string (like "5 minutes") but an expression based on the data itself.

How it works:

Instead of a fixed 5-minute rule for everyone, you can vary the gap based on the input row (e.g., a "Premium" user might get a longer session timeout than a "Guest" user).

Shift in Logic: When gaps are dynamic, the window is no longer just waiting for the "latest input + gap."

The "Union" Rule: Spark treats each event as having its own potential range: [startTime,startTime+dynamicGap]. If these ranges overlap, Spark unions them into a single session.

Note: With dynamic gaps, the closing of the window depends on the collective "coverage" of all events in that session, rather than just a timer starting after the very last event.

3. Implementation Example

In Spark, you would implement the dynamic gap by passing a column expression to the session_window function instead of a hardcoded string.

from pyspark.sql import functions as F

# Assume 'userType' determines the gap

# Premium users get 10 mins, others get 5 mins

dynamic_gap = F.when(events.userType == "premium", "10 minutes") \

.otherwise("5 minutes")

sessionizedCounts = events \

.withWatermark("timestamp", "20 minutes") \

.groupBy(

F.session_window(events.timestamp, dynamic_gap),

events.userId

).count()

This example moves into the most advanced form of windowing in Spark Structured Streaming: Dynamic Session Windows.

In our code, the "inactivity gap" is no longer a constant value for the entire stream. Instead, it is evaluated per row based on the userId.

1. The Concept of "Range Union"

When you use a dynamic gap, Spark calculates a potential range for every single incoming event.

Individual Event Range: For any event i, its range is [timestampi​,timestampi​+gapi​].

The Union Rule: If the range of Event A and the range of Event B overlap, Spark merges them into a single session.

Result: A session window stays open as long as there is a continuous "chain" of overlapping ranges.

2. Code Breakdown

In your specific snippet, the system treats users with different "patience" levels or session requirements:

session_window = session_window(events.timestamp,

sf.when(events.userId == "user1", "5 seconds")# Very short gap

.when(events.userId == "user2", "20 seconds") # Medium gap

.otherwise("5 minutes") # Default long gap

)

How this executes:

User1: If User1 triggers an event at 12:00:00, their session range is 12:00:00 - 12:00:05. If the next event is at 12:00:06, it starts a new session because the ranges don't overlap.

User2: If User2 triggers an event at 12:00:00, their range is 12:00:00 - 12:00:20. They have more "breathing room" to keep the session alive.

Default Users: Anyone else gets a 5-minute gap. Their sessions will likely be much larger and contain many more events.

3. Why "Closing" Logic Changes

As you noted in your text, with dynamic gaps, the closing of the window no longer depends solely on the latest input.

In a static window, you just wait for Last_Event + Fixed_Gap. In a dynamic window, an event that occurred earlier in the sequence might have a massive gap that "covers" later events, or a late event might have a very small gap that doesn't extend the session at all. The window boundary is determined by the maximum extension provided by any event within that merged group.
